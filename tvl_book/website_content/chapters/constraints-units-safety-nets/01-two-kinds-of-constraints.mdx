TVL separates constraints into two categories:

<PredictionPrompt prompt="Before reading the table, decide which rule belongs in the operational-precondition field (`constraints.derived`): `temperature <= 0.8` or `env.context.provider_input_price_usd_per_1k_tokens <= 0.05`.">

The `env.context.*` rule belongs in `constraints.derived` because it depends on the execution environment. `temperature <= 0.8`
is a structural rule because it is a formula over TVARs and can be checked before any trials run.

</PredictionPrompt>

| Category | Evaluated When | Purpose |
|----------|---------------|---------|
| **Structural** | Before trials run (static) | Rules over TVARs that define valid configuration space |
| **Operational Preconditions** | At runtime (dynamic) | Feasibility checks over environment symbols such as provider price, baseline latency, and request headroom |

```yaml
# Fragment: constraints section only (embed in full TVL module)
constraints:
  structural:
    # Checked statically - compiles to SAT/SMT
    - expr: "temperature <= 0.8"
    - when: model = "gpt-4o"
      then: max_tokens >= 1000

  derived:
    # Checked at runtime against environment symbols
    - require: env.context.gateway_baseline_latency_ms <= 250
    - require: env.context.provider_input_price_usd_per_1k_tokens <= 0.05
    - require: env.context.rpm_limit - env.context.rpm_current_load >= 50
```

Operational metrics such as measured latency or token cost belong in measurements, manifests, and runtime
gates. TVL keeps `constraints.derived` scoped to `env.context.*` feasibility checks.

<KnowledgeCheck prompt="Why does TVL keep measured runtime outcomes such as observed latency out of operational preconditions (`constraints.derived`)?">

Because TVL uses `constraints.derived` for feasibility checks over the environment, not for observed experiment
results. Measured latency belongs in objectives, manifests, and rollout evidence so reviewers can distinguish promises
from outcomes.

</KnowledgeCheck>
