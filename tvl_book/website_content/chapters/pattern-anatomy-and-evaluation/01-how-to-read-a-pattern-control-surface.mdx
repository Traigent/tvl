Every pattern page in the atlas answers the same operational questions:

- What scaffold is being introduced?
- Which variables control it?
- Which trade-off does it change?
- Which failure mode appears first?

That common shape is intentional. If patterns are described with incompatible vocabularies, teams end
up comparing narratives instead of comparing governed options.

<ConceptCallout title="What to Look For First" tone="info">

Read the failure modes before you fall in love with the pattern. They usually reveal whether the
pattern is correcting the right problem or simply adding more activity to the system.

</ConceptCallout>

The atlas groups patterns loosely into:

- <Term id="breadth-scaling-patterns">breadth scaling</Term>: more views, experts, or shards
- <Term id="depth-scaling-patterns">depth scaling</Term>: more rounds, critique, or revision

That distinction is not academic. It tells you whether the main budget pressure comes from parallel
coverage or sequential reconsideration.

<KnowledgeCheck prompt="What is the main benefit of describing every pattern with the same control-surface template?">

It makes patterns comparable. Teams can line up variables, trade-offs, and failure modes directly
instead of relying on vague claims that one scaffold is "more advanced" than another.

</KnowledgeCheck>
