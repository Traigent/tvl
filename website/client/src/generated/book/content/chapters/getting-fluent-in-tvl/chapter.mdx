TVL encodes the answers to three questions:

1. **What are we tuning?** (`tvars` — tuned variables with typed domains)
2. **How do we measure success?** (`objectives` and `evaluation_set`)
3. **What rules govern promotion?** (`constraints`, `promotion_policy`)

<ConceptCallout title="How to Learn This Chapter" tone="hint">

- First, read the spec top to bottom until you can point at each section and say what job it does.
- Then run the validation tools so the structure stops feeling abstract.
- Finally, use the microsim to connect YAML choices to visible cost, latency, and quality trade-offs.

</ConceptCallout>

The table below maps each TVL section to its responsibilities.

| Section | Responsibilities |
|---------|------------------|
| `tvl` | Module namespace identifier |
| `tvl_version` | Language version (currently `"1.0"`) |
| `environment` | Snapshot ID plus optional `bindings` and `context` that anchor evaluation |
| `evaluation_set` | Dataset URI and optional seed for reproducibility |
| `tvars` | Tuned variables with explicit types (`bool`, `int`, `float`, `enum[str]`) and domains |
| `constraints` | Structural rules and operational preconditions (`constraints.derived`) |
| `objectives` | Metrics to maximize/minimize; `metric_ref` names the evaluator that computes each one |
| `promotion_policy` | The rollout rulebook: effect sizes, statistical settings, and optional confidence-based release gates |
| `exploration` | Search strategy, budgets, and convergence criteria |
